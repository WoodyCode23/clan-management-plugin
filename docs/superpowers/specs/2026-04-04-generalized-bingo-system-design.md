# Generalized Bingo System Design

## Overview

A fully customizable bingo event system for the Clan Management RuneLite plugin. Admins define board grid size, tile tasks (drop/kc/xp), teams, scoring thresholds, bounties, and drop whitelists in a dedicated Google Sheet. The plugin reads everything dynamically and renders the board, standings, and bounty alerts. Backwards-compatible with the existing SWB26 bingo sheet via a migration function.

## Requirements

- **Fully customizable**: Grid size, tiles, teams, scoring rules — all defined in the sheet
- **Three tile types**: `drop` (whitelist-based), `kc` (boss kill count via WOM), `xp` (skill XP via WOM)
- **Dual standings**: Team standings + individual player standings, always both
- **Enhanced bounties**: Timed alerts (hint + release) in chat/Discord, bonus points awarded to winning team/player, persistent fired state (no duplicate alerts)
- **Separate Google Sheet**: Bingo has its own sheet and Apps Script deployment, independent from the clan management sheet
- **Admin controls**: Setup script in sheet + in-plugin editing for tiles, teams, bounties, manual progress
- **Admin-assigned teams**: Players assigned to teams by admin in Roster, not self-selected
- **SWB26 backwards-compatible**: Migration function converts existing SWB26 sheet to new format
- **Plugin pushes WOM data**: For KC/XP tiles, plugin fetches WOM gains and POSTs to sheet. Sheet is single source of truth.

## Google Sheet Structure

Created by `setupBingoSheet()` in `BingoSetup.gs`.

### Config Tab

Key-value settings:

| Key | Example | Description |
|-----|---------|-------------|
| `gridRows` | `6` | Number of rows |
| `gridCols` | `6` | Number of columns |
| `eventName` | `Solus Winter Bingo 2026` | Display name |
| `startDate` | `2026-03-06T12:00` | Event start (EST) |
| `endDate` | `2026-03-20T12:00` | Event end (EST) |
| `apiKey` | `changeme` | Read API key |
| `adminKey` | `changeme-admin` | Admin API key |

### Tiles Tab

One row per tile:

| Column | Description |
|--------|-------------|
| Code | Tile identifier (e.g., A1, B3) |
| Name | Task description |
| Type | `drop`, `kc`, or `xp` |
| Metric | WOM metric name (for kc/xp tiles, e.g., `zulrah`, `mining`). Blank for drop tiles. |
| Threshold | Points needed for "bingo" (50% completion color) |
| Max | Points for 100% completion |
| Row | Grid row (0-indexed, auto-generated by setup) |
| Col | Grid column (0-indexed, auto-generated by setup) |

### Teams Tab

| Column | Description |
|--------|-------------|
| Code | Short identifier (e.g., WOO, MB) |
| Name | Full team name |

### Roster Tab

| Column | Description |
|--------|-------------|
| RSN | Player display name |
| Team | Team code from Teams tab |

### Whitelist Tab

For `drop`-type tiles only:

| Column | Description |
|--------|-------------|
| Item | Item name (must match RuneLite drop name) |
| Tile Code | Which tile this drop counts toward |
| Points | Points awarded per drop |

### Droplog Tab

Auto-populated by plugin:

| Column | Description |
|--------|-------------|
| Timestamp | When drop was logged |
| RSN | Player name |
| Team | Player's team code |
| Item | Item name |
| Tile | Tile code |
| Points | Points awarded |

### Bounties Tab

| Column | Description |
|--------|-------------|
| Number | Bounty ID (1, 2, 3...) |
| Description | What the bounty is |
| Release Time | When bounty goes live (EST, ISO format) |
| Points | Bonus points awarded to winner |
| Winner | Winning team code or player RSN (blank until awarded) |
| Hint Fired | `TRUE`/`FALSE` — persisted to prevent duplicate alerts |
| Release Fired | `TRUE`/`FALSE` — persisted to prevent duplicate alerts |

### Team Progress Sheets

One sheet per team (auto-created from Teams tab), named by team code (e.g., `WOO`, `MB`):

| Column | Description |
|--------|-------------|
| Tile Code | Matches Tiles tab |
| Tile Name | Copied from Tiles tab |
| Points | Current points accumulated for this tile |

## API Endpoints (BingoAPI.gs)

Single Apps Script deployment. All requests require `key` query parameter.

### GET Endpoints

| Action | Parameters | Returns |
|--------|-----------|---------|
| `ping` | — | `{status: "ok"}` |
| `getBingoConfig` | — | Full config: grid size, event name, dates, all tiles (with type/metric/threshold/row/col), all teams, all bounties |
| `getTeamProgress` | `team` | Per-tile points for the specified team |
| `getAllStandings` | — | All teams ranked by total points (including bounty bonuses) + all players ranked by individual points |
| `getDroplog` | `team` (optional), `limit` (optional) | Recent drops, optionally filtered by team |
| `getWhitelist` | — | All whitelist items with tile codes and points |
| `getBounties` | — | Full bounty list with schedule, points, winners, fired state |

### POST Endpoints

`submitDrop` and `updateTileProgress` require regular `key`. All `admin*` actions require `adminKey`.

| Action | Payload | Purpose |
|--------|---------|---------|
| `submitDrop` | `{player, item, team, timestamp}` | Log a drop (validates against whitelist, awards points). Regular key only. |
| `updateTileProgress` | `{team, tileCode, points}` | Update KC/XP tile progress (called by plugin with WOM data). Regular key only. |
| `adminUpdateTile` | `{tileCode, name?, type?, metric?, threshold?, max?}` | Edit tile properties |
| `adminAddTile` | `{code, name, type, metric, threshold, max, row, col}` | Add a new tile |
| `adminRemoveTile` | `{tileCode}` | Remove a tile |
| `adminUpdateTeam` | `{code, name?}` | Edit or add a team |
| `adminRemoveTeam` | `{code}` | Remove a team |
| `adminUpdateRoster` | `{rsn, team}` | Assign player to team |
| `adminRemoveRoster` | `{rsn}` | Remove player from roster |
| `adminUpdateBounty` | `{number, description?, releaseTime?, points?, winner?}` | Edit bounty or set winner |
| `adminAddBounty` | `{number, description, releaseTime, points}` | Add a new bounty |
| `adminRemoveBounty` | `{number}` | Remove a bounty |
| `adminManualProgress` | `{team, tileCode, points}` | Manually set points for a tile/team |
| `adminUpdateConfig` | `{key, value}` | Update a config value |
| `adminUpdateWhitelist` | `{action: "add"/"remove", item, tileCode, points?}` | Add/remove whitelist items |
| `markBountyFired` | `{number, field: "hintFired"/"releaseFired"}` | Persist bounty alert state. Regular key only. |

## Plugin Architecture

### New Files

**`BingoService.java`** — Singleton HTTP service. Handles all BingoAPI.gs communication. Caches config (refreshed every 5 minutes), fetches progress on normal refresh cycle. Methods: `fetchBingoConfig()`, `fetchTeamProgress(team)`, `fetchAllStandings()`, `submitDrop(...)`, `updateTileProgress(...)`, admin methods.

**`BingoConfig.java`** — Immutable data class holding parsed bingo config: grid dimensions, event name, dates, list of tiles, list of teams, list of bounties. Rebuilt on each config fetch.

**`BingoTile.java`** — Tile model: code, name, type (drop/kc/xp), metric, threshold, max, row, col. Separate from progress data.

**`BingoTeam.java`** — Team model: code, name, totalPoints, rank, bountyBonus.

**`BingoPanel.java`** — Swing panel added as "Bingo" tab. Renders dynamic grid, tile detail on click, team standings, individual standings, bounty section, drop log. All driven by BingoConfig + progress data.

**`BountyScheduler.java`** — Reads bounty schedule from BingoConfig. Manages countdown timers. Fires hint (configurable minutes before) and release alerts in game chat. Posts to Discord. Persists fired state to sheet via `markBountyFired`. On startup: skips past bounties, only schedules future ones. If multiple become eligible simultaneously, fires sequentially with 5-second gaps.

**`BingoSetup.gs`** — Google Apps Script. `setupBingoSheet()` creates all tabs with headers, formatting, example data, and auto-generated tile codes based on grid size. `migrateSWB26()` reads existing SWB26 structure and creates new standardized tabs.

**`BingoAPI.gs`** — Google Apps Script. All GET/POST handlers for the bingo API.

### Modified Files

**`ClanManagementPlugin.java`**:
- Inject `BingoService`
- Add `bingoApiUrl` field (from admin config or plugin config)
- On refresh cycle: if bingo URL set, fetch config, check active dates, fetch progress, update BingoPanel
- For KC/XP tiles: fetch WOM data and push to sheet
- Wire admin callbacks for bingo management
- Initialize BountyScheduler when bingo is active

**`ClanManagementConfig.java`**:
- Add `bingoApiUrl` config item (secret, Admin section)

**`AdminPanel.java`**:
- Add "Bingo Management" section: event status, tile editor (select + edit inline), team editor, bounty manager (set winner, award points), manual progress adjustment, roster management

**`ClanPanel.java`**:
- Dynamically add/remove "Bingo" tab based on whether bingo URL is configured and event is active (between start/end dates)

**`DiscordWebhookService.java`**:
- Add bingo embeds: event start, event end (final standings), drop logged (with tile/team context), bounty hint, bounty live, bounty winner, tile completed

## Bingo Panel UI Layout

Top to bottom within the Bingo tab:

1. **Header**: Event name (gold), countdown timer, player's team name
2. **Board Grid**: Dynamic NxN grid. Cells show tile code. Background color = team completion (Gray <5%, Yellow 5-49%, Green 50%+, Black 100%). Click tile to expand detail.
3. **Tile Detail** (expandable): Full task name, type, progress bar, threshold info. Only shown when a tile is clicked.
4. **Team Standings**: Ranked list with points. Player's team highlighted. Shows bounty bonus separately.
5. **Individual Standings**: Ranked list of all players by personal points. Current player highlighted.
6. **Bounty Section**: Next bounty countdown. List of all bounties with status (upcoming/live/completed), winners, points.
7. **Drop Log**: Recent drops with team filter. Searchable whitelist browser.

## Bounty System

### Alert Flow

1. Plugin loads, BountyScheduler reads bounty list from BingoConfig
2. For each bounty: check `hintFired`/`releaseFired` flags from sheet
3. Skip bounties where release time is in the past AND releaseFired is true
4. Schedule future bounties:
   - Hint fires at `releaseTime - hintMinutes` (default 15 min)
   - Release fires at `releaseTime`
5. When an alert fires:
   - Send game chat message: `[Bingo] Bounty #X hint: releasing in ~Y minutes!` or `[Bingo] Bounty #X is NOW LIVE: <description>`
   - Post Discord embed
   - POST `markBountyFired` to sheet to persist the state
6. If multiple bounties become eligible at once (e.g., plugin was offline): fire sequentially with 5-second delay between each

### Bounty Points

- Admin sets winner via admin panel or sheet
- `adminUpdateBounty` with `winner` field
- Points added to the winning team's bonus total (not to any tile)
- Discord embed announces winner and points
- Standings reflect bounty bonus: `#1 Warriors of Olm — 2,450 pts (+100 bounty)`

## Setup Flow

### New Event

1. Admin creates new Google Sheet
2. Opens Extensions > Apps Script
3. Pastes `BingoSetup.gs` and `BingoAPI.gs`
4. Runs `setupBingoSheet()` from script menu
5. Script prompts for grid size (default 5x5), creates all tabs with headers and example data
6. Admin fills in: tile names/types/thresholds, teams, roster, whitelist items, bounty schedule
7. Sets API key and admin key in Config tab
8. Deploys as Web App (Execute as: Me, Access: Anyone)
9. Copies deployment URL
10. In RuneLite admin panel: pastes URL into "Bingo API URL" field
11. Bingo tab appears for all clan members

### SWB26 Migration

1. Admin opens their existing SWB26 Google Sheet
2. Adds `BingoSetup.gs` and `BingoAPI.gs` to the Apps Script project
3. Runs `migrateSWB26()` from script menu
4. Script reads existing structure (team sheets, Board Detail, Item Listing, Droplog, etc.)
5. Creates new standardized tabs alongside old ones (non-destructive)
6. Maps: team sheets -> Team Progress sheets, Board Detail -> Tiles tab, Item Listing -> Whitelist tab, existing drops -> Droplog tab
7. Admin deploys the updated script as Web App
8. Plugin now reads from the new standardized tabs

## WOM Integration for KC/XP Tiles

1. On refresh cycle, plugin checks for tiles with type `kc` or `xp`
2. Groups tiles by WOM metric
3. Calls `womService.fetchGained(metric, "week")` for each unique metric
4. Sums gains per team using Roster data
5. POSTs `updateTileProgress` for each team/tile combination
6. Sheet stores the progress, all clients read from sheet

Only one plugin instance needs to push per cycle. Use the existing alphabetically-first-member election pattern: the plugin only pushes WOM data if the logged-in player is the alphabetically first online roster member for their team.

## Key Design Decisions

1. **Sheet as source of truth** — All state lives in Google Sheets. Plugin is a read/write client. No local persistence needed.
2. **Separate bingo sheet** — Independent from clan management sheet. Own deployment, own API key. Clean separation.
3. **Dynamic everything** — Grid size, tiles, teams, scoring all read from API. Nothing hardcoded in Java.
4. **Persistent bounty state** — Fired flags stored in sheet. No session-only tracking. Prevents duplicate alerts across restarts.
5. **WOM push model** — Plugin pushes KC/XP data to sheet rather than merging client-side. Ensures all clients see consistent data.
6. **Admin-only team assignment** — No self-selection. Admin controls roster via sheet or plugin.
